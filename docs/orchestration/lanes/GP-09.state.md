## Notes / Decisions

- **YELLOW rationale**: Evidence infrastructure is fully built (Cycles 68-69). The M1 doc deliverable (comprehension checklist) is unblocked and ready to draft. Human playtest evidence is still absent but is not required for the checklist draft step. Strings are partially owned (FrostwakeShipTaskActor) but the highest-traffic UI surface (MainMenuWidget) still uses un-namespaced hardcoded strings.
- **Do not finalize translation** before source strings stabilize. `FrostwakeMainMenuWidget.cpp`'s Japanese strings are the current instability risk: they are draft quality (hardcoded, no namespace, no version tracking) and should not be sent for professional localization review until LOCTEXT-wrapped.
- **"Assist" vs. glossary terms**: The life-action prompt `"Assist"` is ambiguous — it covers rescue, containment, and release of players. The glossary defines these as distinct terms. This is the second-highest wording gap after the MainMenuWidget strings.
- **Forbidden terms check**: No forbidden or stale-concept terms (`Dread`, `Hunger`, `Thrall`, `abyss`, `oxygen system`, etc.) were found in any of the source-string files reviewed. The IP boundary is clean in the current UI string set.
- **Scope boundary**: Do not use UI text to promise proximity voice, dedicated server, or server browser — these appear in the glossary as "Feature claim only after implemented." None of the current UI strings do this; maintain that boundary.
- **Reviewer (Localization + Game Design)** must review the comprehension checklist and the one wording change recommendation before they are acted on in C++ or committed as final.
